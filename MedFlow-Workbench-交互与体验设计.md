## 一、设计哲学

### 1.1 三条原则

| 原则 | 含义 | 反例(不做) |
|------|------|------------|
| **永远不让用户等待而无指示** | 任何超过 200ms 的操作必须有视觉反馈 | 点击按钮后界面静止 |
| **每个数字都可追溯** | 用户看到的每个值都能一键找到来源 | 字段值是黑盒,不知道 LLM 从哪句话抽的 |
| **操作可逆、可回溯** | 任何修改都能撤销,任何阶段都能重跑 | 改错了只能重来,无法恢复 |

### 1.2 节奏感:快-慢-快

医疗数据提取是**长时间专注任务**(单病人可能 10-30 分钟),UI 节奏设计遵循:

```
快速浏览(源图/卡片列表) → 慢速精读(OCR文本/抽取表单) → 快速确认(审核/导出)
   ↑ 紧凑、信息密度高        ↑ 宽松、留白多、字号大        ↑ 汇总、一目了然
```

- **快节奏区**(源图网格、病人列表):紧凑间距(12-14px gap),小字号(12-13px),信息密度高
- **慢节奏区**(OCR 文本、合并阅读器、抽取表单):宽松间距(20-24px gap),大字号(14-16px),充足留白
- **确认区**(审核、导出):表格形式,中等密度,关键数据高亮

### 1.3 视觉层级金字塔

```
            ┌──────────────┐
            │  当前操作焦点  │  ← 最亮、最大、有动效(如正在编辑的输入框)
            ├──────────────┤
            │  当前阶段内容  │  ← 正常亮度、玻璃态卡片
            ├──────────────┤
            │  导航与侧边栏  │  ← 半透明、稍暗
            ├──────────────┤
            │  背景与光晕   │  ← 最暗、模糊、氛围层
            └──────────────┘
```

## 二、动效系统

### 2.1 动效原则

1. **有意义,不装饰**:每个动效都要传达状态变化或空间关系,不做纯炫技
2. **自然曲线**:优先 `cubic-bezier(0.4, 0, 0.2, 1)`(Material 标准 ease),不用 linear
3. **短而有感**:微交互 100-200ms,过渡 200-350ms,绝不超过 400ms(除非是进度动画)
4. **可中断**:动画进行中再次触发不卡顿,用 CSS `transition` 而非 JS 动画(除非需要物理感)
5. **尊重 prefers-reduced-motion**:系统设置降低动效时,所有非必要动画关闭

### 2.2 全局动效变量

```css
:root {
  /* 缓动曲线 */
  --ease-out:     cubic-bezier(0.0, 0.0, 0.2, 1);   /* 减速进入 */
  --ease-in:      cubic-bezier(0.4, 0.0, 1, 1);     /* 加速离开 */
  --ease-in-out:  cubic-bezier(0.4, 0.0, 0.2, 1);   /* 标准双向 */
  --ease-spring:  cubic-bezier(0.34, 1.56, 0.64, 1); /* 弹性(微过冲) */
  --ease-smooth:  cubic-bezier(0.25, 0.1, 0.25, 1); /* 平滑(无过冲) */

  /* 时长 */
  --t-instant:    80ms;    /* 悬停反馈、颜色变化 */
  --t-fast:       150ms;   /* 按钮按下、图标切换 */
  --t-normal:     250ms;   /* 面板展开、卡片高亮 */
  --t-slow:       350ms;   /* 页面切换、侧边栏折叠 */
  --t-spa:        200ms;   /* 单页应用视图切换 */
}
```

### 2.3 逐场景动效设计

#### 2.3.1 病人卡片悬停(Hover)

```css
.patient-card {
  transition: transform var(--t-fast) var(--ease-spring),
              border-color var(--t-instant) ease-out,
              background var(--t-instant) ease-out;
}
.patient-card:hover {
  transform: translateY(-2px);   /* 上浮 2px,轻微弹性 */
  border-color: var(--primary);
}
.patient-card:active {
  transform: translateY(0) scale(0.99);  /* 按下回弹 */
}
```

**心理感受**:卡片像有重量的物理对象,悬停时"被托起",点击时"被按下"。2px 上浮 + 弹性曲线让大脑感知到"可交互"。

#### 2.3.2 病人选中态

```
未选中 ──click──► 选中

边框: 1px solid transparent → 1px solid var(--primary)   [150ms ease-out]
背景: transparent → var(--primary-fade)                    [150ms ease-out]
左侧出现 3px 蓝色竖条: width 0 → 3px                       [200ms ease-spring]
```

3px 竖条从左边"长出"的弹性动画,给选中态一个"被锚定"的确定感。

#### 2.3.3 阶段导航条(StageNav)切换

```
当前阶段胶囊:
  背景: var(--primary) 渐变
  文字: 白色
  阴影: 0 4px 12px rgba(99,102,241,0.3)

切换阶段时:
  1. 旧胶囊: 背景淡出 [200ms],文字变灰
  2. 新胶囊: 背景渐入 [200ms],文字变白,阴影渐入
  3. 底部指示条: 从旧位置滑动到新位置 [300ms ease-spring]
```

底部有一条 2px 的渐变指示条,在阶段间平滑滑动,给用户空间方位感(知道"往右走了")。

#### 2.3.4 阶段内容切换(SPA 视图过渡)

```
切换阶段时,StageContent 区域:
  1. 旧内容: opacity 1→0, translateY(0→8px)  [120ms ease-in]
  2. (间隙 30ms, 避免重叠)
  3. 新内容: opacity 0→1, translateY(-8px→0) [200ms ease-out]
```

8px 的垂直位移很小,但足以让大脑感知"内容换了",而非"内容变了"。比纯 fade 更有层次感。

#### 2.3.5 OCR 卡片状态变化

这是全应用最高频的状态变化,需要精心设计:

```
○ 待识别  →  ⏳ 识别中  →  ✓ 完成

○ 待识别:
  圆圈: 1.5px solid var(--text-faint)
  内部: 透明

⏳ 识别中(进入):
  圆圈边框 → 蓝色 [150ms]
  圆圈内出现 spinner(旋转动画,800ms/圈,linear)
  卡片背景: 出现淡蓝色光晕(0→0.15 opacity)[300ms]
  缩略图: 加 2px 蓝色边框 [200ms]

✓ 完成(进入):
  spinner → ✓ 图标 [200ms, scale 0.8→1.0, ease-spring]
  圆圈边框 → 青色(success)[150ms]
  卡片背景光晕: 淡蓝 → 淡青 → 消失 [400ms total]
  字数文字: 从 0 滚动到实际字数 [400ms, ease-out]
```

**字数滚动动画**:字数从 0 滚动到 1245,用 `requestAnimationFrame` 在 400ms 内完成。这比直接显示"1245字"更有"成果感"——用户能感知到"这一页真的识别出了这么多字"。

#### 2.3.6 进度条

```
线性进度条(阶段内):
  高度: 4px
  背景: rgba(255,255,255,0.08)
  填充: linear-gradient(90deg, var(--primary), var(--primary-2))
  过渡: width 变化用 300ms ease-out(不是线性,让进度"追上来"的感觉)
  
  填充末端有 2px 亮光:
    box-shadow: 0 0 8px var(--primary)
    表示"正在推进"
```

进度条的 `width` 过渡用 300ms 而非瞬时,是因为 WebSocket 消息可能有几百毫秒间隔,平滑过渡让进度条看起来"持续在动"而非"一跳一跳"。

#### 2.3.7 状态点脉冲(Running)

```css
.status-dot.running {
  background: var(--primary);
  animation: pulse 1.4s infinite;
}

@keyframes pulse {
  0%   { box-shadow: 0 0 0 0   var(--primary-fade); }
  70%  { box-shadow: 0 0 0 8px transparent; }
  100% { box-shadow: 0 0 0 0   transparent; }
}
```

1.4s 周期的涟漪扩散,像心跳。选择 1.4s 而非常见的 1s 或 2s,是因为 1s 太急促(焦虑感),2s 太慢(以为卡了),1.4s 接近人在平静状态的心率,传达"正在稳步进行"。

#### 2.3.8 Toast 通知

```
出现(从右上角):
  translateY(-100%) → translateY(0) + opacity 0→1
  [350ms, ease-spring, 微过冲 5%]

停留:
  3.5 秒(成功类) / 5 秒(警告/错误类)
  最后 500 秒开始淡出

消失:
  opacity 1→0 + translateX(20px)
  [250ms, ease-in]

最多同时显示 3 条,超出则最早的自动消失
```

Toast 类型与颜色:

| 类型 | 颜色 | 图标 | 示例 |
|------|------|------|------|
| success | 青色 #22d3ee | ✓ | OCR 完成:12/12 页 |
| info | 蓝色 #6366f1 | ℹ | 开始批量抽取 5 人 |
| warning | 橙色 #f59e0b | ⚠ | 合并文档已被编辑,建议重新抽取 |
| error | 红色 #f87171 | ✗ | OCR 失败:Token 未配置 |

#### 2.3.9 按钮交互

```
默认按钮(.btn):
  背景: var(--glass)
  边框: 1px solid var(--glass-border)
  过渡: all var(--t-instant) ease-out

悬停:
  边框色 → var(--primary)
  文字色 → var(--primary)
  背景微微变亮

按下(:active):
  transform: scale(0.97)       ← 缩小 3%
  过渡: 60ms (极快,模拟物理触碰)

主按钮(.btn-primary):
  背景: linear-gradient(135deg, primary, primary-2)
  悬停: filter brightness(1.08)  ← 渐变变亮,不是变色
  按下: scale(0.97) + brightness(0.95)  ← 变暗一点,模拟"按下去"

禁用:
  opacity: 0.4
  cursor: not-allowed
  悬停无变化
```

#### 2.3.10 面板折叠(StagePanel / PatientSidebar)

```
折叠(宽→窄):
  width: 340px → 0 (或 48px)
  [300ms, ease-in-out]
  内部内容: opacity 1→0 [150ms, 在 width 动画前半段完成]

展开(窄→宽):
  width: 0 → 340px
  [300ms, ease-spring]
  内部内容: opacity 0→1 [200ms, 延迟 100ms 开始, 在 width 动画后半段]
```

内容先消失再折叠,展开时先撑开再显示内容——避免内容在缩放过程中被挤压变形。

#### 2.3.11 拖拽排序(源图页序)

```
拖起:
  被拖元素: scale(1.05) + 阴影增强 + opacity 0.9
  [150ms, ease-spring]
  光标: grabbing

拖动中:
  其他元素: 让出位置时,用 FLIP 技术
    1. 记录旧位置
    2. DOM 移动后记录新位置
    3. 用 transform 从旧位置过渡到新位置
    [200ms, ease-spring]

放下:
  被拖元素: scale(1.05→1.0) + 阴影减弱
  [200ms, ease-spring]
  光标: default

拖拽过程中显示半透明提示条:
  "⚠ 拖拽将影响 OCR 和合并的页顺序"
  出现: opacity 0→1, translateY(-10px→0) [200ms]
```

#### 2.3.12 字段高亮联动(抽取阶段)

点击右侧字段 → 左侧源文本滚动并高亮:

```
1. 源文本容器滚动:
   scrollToLeft(targetElement.offsetLeft - 40px)
   行为: smooth
   时长: 浏览器原生 ~300ms

2. 目标段落高亮:
   背景色: transparent → rgba(99,102,241,0.15)
   [200ms, ease-out]
   
   保持 2 秒后淡出:
   [800ms, ease-in]
   
   如果用户在 2 秒内点击其他字段,立即取消当前高亮,开始新的

3. 被点击字段(右侧):
   左边框: 2px solid var(--primary) 出现 [150ms]
   背景: 微微变亮 [150ms]
   保持高亮直到点击其他字段
```

#### 2.3.13 抽取字段编辑追踪

用户修改字段值后:

```
字段容器:
  左边框: transparent → 3px solid var(--primary) [200ms, ease-spring]
  背景: 微微变亮 [150ms]

字段值文字:
  color: var(--text) → var(--primary) [200ms]
  (蓝色文字表示"人工修改值")

Tooltip(悬停时):
  "✏ 已编辑 · 原值: 80 · 10:45 修改"
  出现: opacity 0→1, translateY(4px→0) [150ms, ease-out]
  延迟 500ms 出现(避免快速划过时闪烁)
```

#### 2.3.14 页面切换动画(合并阅读器翻页)

```
下一页:
  当前页: opacity 1→0, translateX(0→-20px) [150ms, ease-in]
  下一页: opacity 0→1, translateX(20px→0) [200ms, ease-out]
  (两页有 50ms 交叉,避免空白闪烁)

上一页:
  当前页: opacity 1→0, translateX(0→20px) [150ms, ease-in]
  上一页: opacity 0→1, translateX(-20px→0) [200ms, ease-out]

方向感知:
  → 键 / 下一页按钮: 内容向左滑(新内容从右进)
  ← 键 / 上一页按钮: 内容向右滑(新内容从左进)
```

方向一致性:翻页方向与内容进入方向一致,符合"翻书"的心理模型。

#### 2.3.15 Lightbox(全屏图片查看)

```
打开:
  遮罩: opacity 0→1 [200ms]
  图片: scale(0.9→1) + opacity 0→1 [300ms, ease-spring]
  起始位置: 从被点击的缩略图位置开始(FLIP)

关闭:
  图片: scale(1→0.9) + opacity 1→0 [200ms, ease-in]
  遮罩: opacity 1→0 [200ms, 延迟 50ms 开始]
```

从缩略图位置"飞出"到全屏,给用户"这张图被放大了"的连续感,而非"突然弹出一个新东西"。

## 三、逐区域交互细节

### 3.1 TopBar

#### 3.1.1 导入病人按钮

```
[导入病人] 点击行为:
  1. 按钮: 短暂 scale(0.97) [60ms]
  2. 出现下拉面板:
     opacity 0→1 + translateY(-8px→0) [200ms, ease-spring]
     面板宽 320px,含:
       · 文本输入框(粘贴路径)
       · [选择文件夹] 按钮(打开原生对话框)
       · [导入] 确认按钮
  3. 点击外部关闭面板: opacity 1→0 [150ms]
```

#### 3.1.2 全局搜索

```
搜索框聚焦:
  宽度: 200px → 320px [200ms, ease-spring]
  边框色 → var(--primary) [150ms]
  出现下拉提示: "搜索病人名称或字段值"

输入时:
  实时搜索(300ms 防抖)
  下拉显示匹配结果(最多 8 条)
  每条: 病人头像 + 名称 + 匹配阶段 + 匹配上下文

无结果:
  下拉显示 "未找到匹配,试试其他关键词"
```

### 3.2 PatientSidebar

#### 3.2.1 病人卡片点击

```
单击(选中):
  1. 卡片: 选中态动画(见 2.3.2)
  2. 中间区: 加载新病人数据
     · 显示骨架屏(见 6.2)
     · 数据到达后,fade in 内容
  3. StageNav: 切到该病人的当前阶段

Ctrl/Cmd+点击(追加):
  1. 卡片: 选中态动画
  2. 中间区: 切换到批量视图
     · 旧内容 fade out [120ms]
     · 批量表格 fade in [200ms]

Shift+点击(范围):
  1. 从上次选中到当前,中间所有卡片: 依次选中
     · 每张卡片延迟 30ms 触发选中动画
     · 形成从上到下的"波浪"选中效果
     · 总时长不超过 500ms(避免太慢)
```

Shift+选中的 30ms 级联延迟,营造"选择在流动"的感觉,比全部同时选中更有掌控感。

#### 3.2.2 筛选切换

```
点击筛选项(如"失败"):
  1. 旧筛选项: 文字变灰 [150ms]
  2. 新筛选项: 文字变亮 + 左侧出现 2px 竖条 [200ms, ease-spring]
  3. 病人列表:
     · 不匹配的卡片: opacity 1→0.3, 不改变布局 [200ms]
       (不是直接消失,而是变暗,让用户知道"还有这些人,只是被筛选掉了")
     · 匹配的卡片: 保持正常
  4. 如果筛选后超过 20 人,显示 "显示 20/25, [显示全部]"
```

筛选不用 display:none(会引起布局跳动),用 opacity 变暗——保留空间,用户心理上知道"这些人还在"。

#### 3.2.3 导入新病人

```
导入成功后:
  1. 新卡片从顶部"滑入":
     height: 0 → 正常高度 [300ms, ease-spring]
     opacity: 0 → 1 [200ms, 延迟 100ms]
  2. 如果有多个新卡片,级联出现:
     每张延迟 50ms
  3. Toast: "✓ 已导入 3 位病人" [3.5s]
  4. 计数文字: "共 14 位" 滚动更新 [400ms]
```

### 3.3 StageNav

#### 3.3.1 阶段胶囊状态

| 状态 | 视觉 | 说明 |
|------|------|------|
| pending | 灰色文字,无背景 | 未到达 |
| running | 蓝色文字,淡蓝背景,脉冲 | 正在执行 |
| done | 青色文字,淡青背景 | 已完成 |
| error | 红色文字,淡红背景 | 失败 |
| skipped | 灰色文字,虚线边框 | 主动跳过 |
| stale | 橙色文字,橙色背景,⚠ 图标 | 上游已变,需重跑 |
| current(查看中) | 白色文字,渐变背景,阴影 | 当前查看的阶段 |

#### 3.3.2 点击切换阶段

```
点击非当前阶段:
  1. 当前阶段胶囊: 渐变背景淡出 [200ms]
  2. 被点击胶囊: 渐变背景淡入 [200ms]
  3. 底部指示条: 滑动到新位置 [300ms, ease-spring]
  4. StageContent: 视图切换动画(见 2.3.4)
  5. StagePanel: 右侧面板内容更新
     · 旧内容 fade out [120ms]
     · 新内容 fade in [200ms]
```

### 3.4 StagePanel(右侧操作面板)

#### 3.4.1 执行按钮

```
[▶ 执行此阶段] 按钮:

未运行时:
  正常主按钮样式

点击:
  1. scale(0.97) [60ms]
  2. 文字: "▶ 执行此阶段" → "⏳ 正在执行..." [200ms, crossfade]
  3. 按钮变禁用,出现下方进度条
  4. 按钮左侧出现 spinner(白色,16×16)

运行中:
```

## 四、状态反馈系统

### 4.1 反馈层级

| 层级 | 响应时间 | 示例 |
|------|---------|------|
| 即时(< 100ms) | 按钮按下、悬停、焦点变化 | 用户不觉得"有延迟" |
| 快速(100-300ms) | 卡片选中、面板展开、状态切换 | 用户感知"流畅" |
| 正常(300-600ms) | 页面切换、数据加载完成 | 用户感知"在响应" |
| 延迟(> 600ms) | OCR、LLM 抽取等长操作 | 必须有进度指示或骨架屏 |

### 4.2 视觉反馈清单

每个用户操作**至少**有以下反馈之一:

| 操作 | 反馈 |
|------|------|
| 点击按钮 | scale(0.97) + 颜色变化 |
| 点击卡片 | 选中态(边框+背景+竖条) |
| 悬停可点击元素 | cursor: pointer + 颜色/边框变化 |
| 输入框聚焦 | 边框变色 + 光晕 |
| 输入框值变化 | "未保存"标记 |
| 保存成功 | 短暂青色闪烁 + "已保存" |
| 验证失败 | 红色边框 + 抖动 |
| 数据加载中 | 骨架屏或 spinner |
| 长操作进行中 | 进度条 + 实时日志 |
| 操作完成 | Toast 通知 |
| 操作失败 | 红色 Toast + 错误详情 |

### 4.3 声音反馈(可选,默认关闭)

设置中可开启:
- OCR 单页完成:轻微"叮"(不刺耳)
- 抽取完成:柔和提示音
- 错误发生:低沉"嘟"

音量很轻,只在后台执行时提醒用户"回来看看"。

## 六、加载与等待体验

### 6.1 加载策略

| 场景 | 策略 | 理由 |
|------|------|------|
| 病人列表 | 骨架屏(卡片形状) | 列表结构固定,骨架屏最自然 |
| 阶段详情 | 骨架屏 + 微 spinner | 内容结构已知 |
| 图片缩略图 | 渐进式加载(模糊→清晰) | 图片有明确的加载过程 |
| OCR 文本 | 直接显示(已有数据) | 文本是本地文件,读取快 |
| LLM 抽取 | 进度条 + 实时日志 | 长操作,需要持续反馈 |
| 导出 | 进度条 | 中等时长 |

### 6.2 骨架屏设计

```
┌──────────────────────────┐
│ ┌────┐ ┌──────────────┐  │
│ │ ▓▓▓│ │▓▓▓▓▓▓▓▓▓▓▓▓ │  │  ← 灰色 shimmer 块
│ │ ▓▓▓│ │▓▓▓▓▓▓▓▓     │  │
│ └────┘ └──────────────┘  │
│                          │
│ ┌────────────────────┐   │
│ │ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │   │
│ │ ▓▓▓▓▓▓▓▓▓▓▓       │   │
│ └────────────────────┘   │
└──────────────────────────┘

shimmer 动画:
  background: linear-gradient(90deg, rgba(255,255,255,0.05) → rgba(255,255,255,0.1) → rgba(255,255,255,0.05))
  background-size: 200% 100%
  animation: shimmer 1.5s infinite

@keyframes shimmer {
  0%   { background-position: 200% 0; }
  100% { background-position: -200% 0; }
}
```

1.5s 的 shimmer 周期——慢于状态点脉冲(1.4s)但视觉上是"光在流动"而非"在心跳",区分"加载中"和"执行中"。

### 6.3 图片渐进加载

```
1. <img> src = 缩略图 URL
2. 同时加载高清图(在内存中)
3. 高清图加载完成:
   · 缩略图淡出 [200ms]
   · 高清图淡入 [300ms]
   · 或者:用 <canvas> 实现从模糊到清晰的渐进解码
```

### 6.4 LLM 抽取等待

LLM 抽取通常需要 10-30 秒,是最长的等待:

```
点击 [▶ 执行抽取] 后:
  1. 按钮变 spinner + "⏳ 正在抽取..."
  2. StageContent 出现等待视图:
     ┌────────────────────────────────┐
     │                                │
     │        🧠                      │
     │   AI 正在阅读病历...            │
     │                                │
     │   ▓▓▓▓▓▓▓▓░░░░░░░ 60%          │
     │   正在提取实验室检查字段...      │
     │                                │
     │   已耗时: 18 秒                 │
     │   预计还需: 12 秒               │
     │                                │
     └────────────────────────────────┘
  
  3. 右侧面板实时日志:
     10:32 发送请求到 DeepSeek
     10:32 病历内容: 15678 字
     10:34 收到响应,正在解析...
     10:34 ✓ 解析完成,80 个字段

     · 等待视图 fade out [200ms]
     · 抽取结果表单 fade in [300ms]

```

"已耗时"和"预计还需"让用户有预期,减少焦虑。

### 6.5 OCR 等待

每页完成后,卡片自动滚动到下一张待识别卡片(如果用户在"跟随模式")。

## 七、错误与空状态

### 7.1 错误状态设计原则

1. **说人话**:不用 "Exception at line 42",用 "OCR 接口未响应,请检查网络"
2. **给解决方案**:不仅说错误,还告诉用户怎么办
3. **可重试**:错误状态都有 [重试] 按钮
4. **不甩锅**:不弹出浏览器 alert,用应用内错误面板

### 7.2 错误展示

**卡片级错误(病人卡片):**

```
┌──────────────────────┐
│ ✗  赵六               │
│   OCR失败             │
│   ⚠ Token未配置       │  ← 红色文字
│   [配置Token] [重试]  │  ← 内联操作按钮
└──────────────────────┘

错误文字: 12px,红色
操作按钮: 11px,边框样式(非主按钮)
```

**阶段级错误(StageContent):**

```
┌────────────────────────────────────────┐
│                                        │
│            ✗                            │
│                                        │
│      OCR 执行失败                       │
│      Token 未配置                       │
│                                        │
│   原因: OCR API 需要 Token 鉴权,       │
│   当前未设置。请在右侧配置 Token。      │
│                                        │
│   [配置 Token]  [重试]  [查看日志]      │
│                                        │
└────────────────────────────────────────┘

✗ 图标: 48px, 红色, scale(0→1) [400ms, ease-spring]
文字: 标题 18px, 描述 14px

```

### 7.3 空状态

**无病人时(PatientSidebar):**

```
│                          │
```

**无数据时(各阶段):**

### 7.4 网络断开

```
WebSocket 断开时:
  StatusBar: 出现 "⚠ 连接已断开,正在重连..." (橙色)

  重连失败 3 次: "✗ 无法连接后端,请检查服务" (红色) + [重试] 按钮

断开期间:
  用户仍可查看已加载的数据(缓存)
  操作按钮禁用 + tooltip "连接断开,操作不可用"
```

## 八、认知负荷管理

### 8.1 渐进式信息披露

**原则:默认只展示必要信息,细节按需展开。**

| 场景 | 默认 | 展开 |
|------|------|------|
| OCR 卡片 | 标题 + 状态 + 字数 | 点击展开全文 |
| 抽取字段 | 字段名 + 值 | 悬停看原始值;点击展开备注 |
| 右侧面板 | 配置 + 进度 + 日志 | 配置抽屉可折叠 |
| LLM 响应 | 不显示 | 点击 [查看原始响应] 展开 |
| 编辑历史 | 不显示 | 点击 [查看编辑历史] 展开 |

### 8.2 视觉降噪

- **非焦点内容降低对比度**:未选中的病人卡片、非当前阶段,文字颜色降一档
- **数量限制**:OCR 搜索结果最多显示 50 个匹配,超出显示"显示前 50 个"
- **延迟出现**:Tooltip 延迟 500ms 出现,避免快速移动鼠标时闪烁
- **分隔线用渐变**:不用硬边框,用 `linear-gradient(transparent → border → transparent)` 的渐变线,视觉更柔和

### 8.3 操作确认层级

| 操作影响 | 确认方式 |
|---------|---------|
| 低(查看、展开) | 无确认 |
| 中(编辑保存、单页重 OCR) | 内联确认(按钮变色)或轻确认气泡 |
| 高(重新执行阶段、覆盖编辑) | 对话框确认 |
| 极高(删除病人、清空所有产物) | 对话框 + 输入病人名称确认 |

**轻确认气泡(inline popover):**

```
点击 [↻ 重新OCR此页]:
  按钮下方弹出气泡:
  ┌─────────────────────────┐
  │ 重新识别将覆盖当前文本   │
  │ [确定] [取消]           │
  └─────────────────────────┘
  不遮挡其他内容,点击外部关闭
```

**对话框(重要操作):**

```
点击 [↻ 重新执行抽取]:
  ┌──────────────────────────────────────┐
  │ ⚠ 重新执行抽取                       │
  ├──────────────────────────────────────┤
  │ 这将清除当前的抽取结果,包括你对      │
  │ 5 个字段的人工修改。此操作不可撤销。 │
  │                                      │
  │ [取消]  [确认重新执行]                │
  └──────────────────────────────────────┘
  
  出现: scale(0.9→1) + opacity 0→1 [250ms, ease-spring]
  遮罩: opacity 0→0.5 [200ms]
```

### 8.4 防误操作

- **按钮防抖**:点击后 500ms 内重复点击无效
- **保存前提示**:有未保存修改时切换阶段,提示"有未保存的修改,是否保存?"
- **删除二次确认**:删除病人需输入名称或点击两次

## 九、情感化设计

### 9.1 空间感与"反重力"氛围

延续 antigravity 的玻璃态 + 光晕设计:

```css
/* 底层光晕,缓慢移动 */
.app-root::before {
  content: "";
  position: fixed;
  inset: -20%;
  background:
    radial-gradient(circle at 20% 20%, var(--bg-glow-1), transparent 45%),
    radial-gradient(circle at 80% 70%, var(--bg-glow-2), transparent 50%);
  animation: glow-drift 30s ease-in-out infinite alternate;
  pointer-events: none;
}

@keyframes glow-drift {
  0%   { transform: translate(0, 0) scale(1); }
  100% { transform: translate(-3%, 2%) scale(1.05); }
}
```

30 秒周期的光晕缓慢漂移,几乎不可察觉,但营造出"空间是活的"氛围——不是死板的固定背景,而是有微妙呼吸感的空间。选择 30s 是因为低于 20s 会被人眼察觉到运动(分心),高于 40s 则失去意义。

### 9.2 完成感

**单病人全流程完成时:**

```
最后一个阶段(导出)完成:
  1. 病人卡片: 状态点 → 青色 + 出现 ✓ 图标 [400ms, ease-spring]
  2. 卡片短暂"庆祝":
     scale(1→1.03→1) [600ms, ease-spring]
     出现一圈青色光晕(从中心扩散,opacity 1→0) [800ms]
  3. Toast: "✓ 张三 已完成全部流程" [3.5s]
  4. StatusBar 统计更新: "已完成" +1 [数字滚动 400ms]
```

克制而温暖的确认——不是满屏彩带,而是一个微妙的"呼吸"和光晕,传达"这件事做好了"。

### 9.3 人物感

- **头像**:每个病人有基于名称哈希的颜色,让病人不只是文字,而是有"身份色"的对象
- **进度的人格化**:不显示"OCR 67%",显示"OCR 8/12 页"——用可数的单位,让进度更具体
- **时间标注**:日志带时间戳,让用户感知"这件事是在 10:33 发生的",有过程感

### 9.4 微文案

文案是用户体验的隐性部分,所有文案遵循**准确、温和、不甩锅**:

| 场景 | ❌ 不要 | ✅ 要 |
|------|---------|-------|
| OCR 失败 | "OCR 失败:错误码 500" | "OCR 接口未响应,可能是网络问题或 Token 过期。[测试连接]" |
| 必填为空 | "此字段必填" | "这个字段还没有值,如果病历中未提及,请填 -1" |
| 未保存离开 | "确定离开?" | "你有未保存的修改(第 3 页),离开前要保存吗?" |
| 删除病人 | "确认删除?" | "删除后无法恢复,该病人的所有中间产物都会清除。确定删除赵六吗?" |
| 导入无图 | "未找到图片" | "这个文件夹里没有图片文件,请确认选择的是病历图片所在目录" |
| 加载中 | "Loading..." | "正在读取 OCR 结果..." |
| LLM 超时 | "请求超时" | "大模型响应超时(30秒),可能是网络慢或模型繁忙。[重试] 或 [换一个模型]" |

## 十、可访问性与健壮性

### 10.1 键盘可达性

- 所有交互元素可通过 Tab 到达
- 焦点可见:`:focus-visible` 有明显边框(2px solid var(--primary))
- 焦点顺序符合视觉顺序(从左到右,从上到下)
- 不会出现"键盘陷阱"(Tab 进去出不来)

### 10.2 颜色对比度

- 主文字 `--text (#e6eaf5)` 对背景 `--bg (#0b0f1a)`:对比度 14.8:1(远超 WCAG AAA 7:1)
- 次文字 `--text-subtle (#8b93a7)`:对比度 6.3:1(达 WCAG AA 4.5:1)
- 淡文字 `--text-faint (#5b6376)`:对比度 3.2:1(仅用于非关键信息)

状态颜色不依赖色觉辨别,同时有形状/图标区分:
- pending: 灰圆点
- running: 蓝脉冲圆点
- done: 青圆点 + ✓
- error: 红圆点 + ✗

### 10.3 响应式

最小窗口尺寸 1024×640,低于此尺寸时:
- PatientSidebar 折叠为 48px(只显示头像)
- StagePanel 折叠为按钮组
- StageContent 占满

### 10.4 数据安全

- **自动保存**:编辑后 3 秒无操作自动保存草稿到 localStorage(防止崩溃丢失)
- **离开提示**:有未保存修改时,关闭窗口提示
- **崩溃恢复**:Electron 重启后,从 state.json 恢复所有病人状态
- **编辑历史**:每次保存记录 {timestamp, field, old_value, new_value},可查看完整修改历史

### 10.5 性能感知

| 场景 | 策略 |
|------|------|
| 100+ 病人 | 虚拟滚动(@tanstack/react-virtual) |
| 50+ 页 OCR | 卡片虚拟滚动,只渲染可见的 + 上下各 3 张 |
| 超长合并文本 | 分页加载,每页 20KB,虚拟滚动 |
| 大图缩略图 | Intersection Observer 懒加载 |
| 频繁 WebSocket | 消息合并(100ms 内的多条进度合并为一次渲染) |
